# Обмеження токенів контексту OpenAI

У попередньому уроці ми додали **контекст (історію повідомлень)** — на кожному запиті надсилаємо всю розмову, щоб модель «пам'ятала» попередні повідомлення.

Це працює добре для осмисленого діалогу, але з часом контекст **нескінченно зростає** — особливо якщо відповіді довгі (наприклад, есе на кілька сотень слів). Кожна нова відповідь знову потрапляє в чергу і знову надсилається в API.

## Навіщо обмежувати контекст

Дві причини:

1. **Вартість** — OpenAI тарифікує за кількістю токенів; чим більший контекст, тим дорожчі запити.
2. **Ліміт моделі** — у кожної моделі є максимальний розмір контекстного вікна, наприклад:
   - GPT-3.5-turbo — ~4 000 токенів;
   - новіші моделі — ~16 000 токенів і більше.

Тому потрібен **ліміт для черги повідомлень** і механізм скорочення контексту, коли він перевищує поріг.

## Що реалізуємо

1. Константу **`MAX_TOKENS`** (для тестів ставимо низьке значення).
2. Підрахунок токенів через пакет **`tiktoken`** (`encoding_for_model`).
3. Перевірку розміру контексту після кожного запиту.
4. Функцію **`deleteOlderMessages`**, яка видаляє найстаріші повідомлення, поки контекст не вкладеться в ліміт.

## Константа ліміту та tiktoken

```typescript
import { encoding_for_model } from "tiktoken";

const encoder = encoding_for_model("gpt-4o-mini");
const MAX_TOKENS = 700;
```

> Більшість моделей використовують однакове кодування, тому вибір моделі для енкодера не критичний.

## Перевірка після відповіді API

Після `chat.completions.create` у відповіді є необов'язкове поле **`usage`** з кількістю токенів:

```typescript
if (response.usage && response.usage.total_tokens > MAX_TOKENS) {
  deleteOlderMessages();
}
```

Після перевищення ліміту `deleteOlderMessages` перераховує контекст через `getContextLength` і видаляє найстаріші повідомлення.

## Функція `getContextLength`

Рахує загальну кількість токенів у масиві `context`. Контент повідомлення в TypeScript може бути:

- `string`;
- масив `ChatCompletionContentPart[]`;
- `null` / `undefined`.

```typescript
function getContextLength() {
  let length = 0;
  context.forEach((message) => {
    if (typeof message.content == "string") {
      length += encoder.encode(message.content).length;
    } else if (Array.isArray(message.content)) {
      message.content.forEach((messageContent) => {
        if (messageContent.type == "text") {
          length += encoder.encode(messageContent.text).length;
        }
      });
    }
  });
  return length;
}
```

## Функція `deleteOlderMessages`

Ітеративний процес:

```typescript
function deleteOlderMessages() {
  let contextLength = getContextLength();
  while (contextLength > MAX_TOKENS) {
    for (let i = 0; i < context.length; i++) {
      const message = context[i];
      if (message.role != "system") {
        context.splice(i, 1);
        contextLength = getContextLength();
        console.log("New context length: " + contextLength);
        break;
      }
    }
  }
}
```

Ключові моменти:

- **`while`** — повторюємо, поки контекст перевищує ліміт (одного видалення може бути недостатньо).
- **`for` з індексом від 0** — гарантуємо видалення **найстаріших** повідомлень (у `for...of` порядок не завжди передбачуваний).
- **`system`-роль не чіпаємо** — системний промпт завжди залишається в контексті.
- **`splice`** — видаляє повідомлення з масиву на місці.

## Демонстрація в debug-режимі

1. Запускаємо застосунок у режимі налагодження (breakpoint у `deleteOlderMessages`).
2. Запит: «Tell me about the First Punic War in 200 words» — отримуємо відповідь.
3. Запит: «Tell me about the Second Punic War» — breakpoint спрацьовує.

Що відбувається:

- Контекст: `system` → `user` (перше питання) → `assistant` (перша відповідь) → `user` (друге питання).
- Перша ітерація: пропускаємо `system`, видаляємо **перше user-повідомлення** (про First Punic War).
- Контекст ще ~735 токенів — більше ліміту.
- Друга ітерація: видаляємо **першу assistant-відповідь**.
- Контекст ~377 токенів — менше ліміту, цикл завершується.
- Запит іде з **скороченим контекстом**, відповідь все одно приходить.

## Повний код `src/basic-chat/index.ts`

```typescript
import OpenAI from "openai";
import { encoding_for_model } from "tiktoken";

const openai = new OpenAI();
const encoder = encoding_for_model("gpt-4o-mini");

const MAX_TOKENS = 700;

const context: OpenAI.Chat.Completions.ChatCompletionMessageParam[] = [
  {
    role: "system",
    content: "You are a helpful chatbot",
  },
];

async function createChatCompletion() {
  const response = await openai.chat.completions.create({
    model: "gpt-4o-mini",
    messages: context,
  });
  const responseMessage = response.choices[0].message;
  context.push(responseMessage);

  if (response.usage && response.usage.total_tokens > MAX_TOKENS) {
    deleteOlderMessages();
  }

  console.log(
    `${response.choices[0].message.role}: ${response.choices[0].message.content}`,
  );
}

process.stdin.addListener("data", async function (input) {
  const userInput = input.toString().trim();
  context.push({
    role: "user",
    content: userInput,
  });
  await createChatCompletion();
});

function deleteOlderMessages() {
  let contextLength = getContextLength();
  while (contextLength > MAX_TOKENS) {
    for (let i = 0; i < context.length; i++) {
      const message = context[i];
      if (message.role != "system") {
        context.splice(i, 1);
        contextLength = getContextLength();
        console.log("New context length: " + contextLength);
        break;
      }
    }
  }
}

function getContextLength() {
  let length = 0;
  context.forEach((message) => {
    if (typeof message.content == "string") {
      length += encoder.encode(message.content).length;
    } else if (Array.isArray(message.content)) {
      message.content.forEach((messageContent) => {
        if (messageContent.type == "text") {
          length += encoder.encode(messageContent.text).length;
        }
      });
    }
  });
  return length;
}
```

## Підсумок

У цьому уроці ми навчилися **контролювати розмір черги повідомлень**:

- рахуємо токени через `tiktoken`;
- задаємо `MAX_TOKENS`;
- ітеративно видаляємо найстаріші user/assistant-повідомлення;
- зберігаємо system-промпт.

Це зменшує вартість API-викликів і запобігає перевищенню ліміту моделі, ціною втрати частини ранньої історії діалогу.
