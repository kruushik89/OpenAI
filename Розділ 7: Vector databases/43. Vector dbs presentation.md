# Vector databases: навіщо потрібні і як працюють

Головний виклик при розробці AI-застосунків — **ефективна обробка даних**. У попередніх розділах embeddings зберігали в JSON: навіть для кількох записів файл ставав дуже великим. Такий підхід **не масштабується** і не підходить для реального світу.

Рішення — **vector database**.

## Що робить vector DB

Коротко: зберігає **вектори (embeddings)**, згенеровані моделлю.

### Порівняння з SQL

| | SQL / традиційна БД | Vector DB |
|--|---------------------|-----------|
| **Запит** | за **точним збігом** значень | за **similarity** (схожістю) |
| **Результат** | записи, де значення збігаються 1:1 | найближчі / найсхожіші записи |

Є search term / запит, але результати повертаються **не за exact match**, а за **близькістю векторів**.

### Дві головні можливості

1. Зберігають **дані + embeddings**.
2. Приймають **embedding як параметр запиту** і допомагають з **similarity search**.

Під капотом — оптимізовані алгоритми **Approximate Nearest Neighbor (ANN)** search.

---

## Як це працює (схема)

### Додавання даних

1. Беремо контент (або релевантну частину).
2. Пропускаємо через **embedding model** → отримуємо вектор.
3. Зберігаємо embedding **разом з іншими даними** у vector DB.

### Запит

1. Search term / query також пропускаємо через **embedding model** → вектор запиту.
2. За цим вектором робимо query до БД.
3. БД повертає результат (найрелевантніші записи).

Практику цього побачимо далі в розділі.
